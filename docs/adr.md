# Architecture Decision Records

本ドキュメントはプロジェクトにおける技術選定・設計判断の背景を記録する。
新たな意思決定が発生した場合は本ファイルに追記する。

## ADR-001: プログラミング言語として Rust を選択

**日付**: 2025年

**ステータス**: 採用

**背景**

個人用家計簿アプリの開発にあたり、学習目的も兼ねた言語選定が必要だった。

**検討した選択肢**

- Python: 習熟済みのため実装コストは低いが学習価値が薄い
- Go: CLIツール開発と親和性が高く有力候補だった
- Rust: 所有権・借用など他言語にない概念を持ち、学習価値が高い

**決定理由**

- 所有権・借用・ライフタイムという概念を実践で習得したい
- 静的型付けによる安全性をコンパイル時に担保できる
- CLIツールからAPIサーバーまで同一言語で実装できる
- 将来的にシステムプログラミングへの応用が可能

**トレードオフ**

- 学習コストが高く、初期の実装速度は遅くなる
- エコシステムが Python・Go と比べて小さい場面がある

## ADR-002: DB として SQLite（rusqlite）を選択

**日付**: 2025年

**ステータス**: 採用（フェーズ2で Turso に移行済み）

**背景**

インメモリDBからファイル永続化まで対応できるDBが必要だった。

**検討した選択肢**

- `Vec<Transaction>` + `HashMap`: 依存ゼロだがセッション限定
- SQLite（rusqlite）: ファイル永続化が可能で学習コストが低い
- PostgreSQL: 本格的なRDBMSだが個人用途では過剰

**決定理由**

- 家計簿として実用するためデータの永続化が必須
- インメモリモードで開発・テストし、同一コードでファイルDBに移行できる
- `bundled` featureによりシステムへのSQLiteインストールが不要
- フェーズ2でのクラウド移行先（Turso）がSQLite互換のため移行コストが低い

**トレードオフ**

- Cloud Run 環境ではファイルが永続化されないためフェーズ2で移行が必要

## ADR-003: CLIパーサーとして clap v4 を選択

**日付**: 2025年

**ステータス**: 採用

**背景**

`add`, `list`, `edit`, `delete`, `summary`, `progress`, `budget` など複数のサブコマンドを持つCLIの実装が必要だった。

**検討した選択肢**

- `std::env::args()` による手実装: 依存ゼロだがサブコマンドの増加で限界が来る
- `argh`: 軽量だが機能が少ない
- `clap` v4: Rustのデファクトスタンダード

**決定理由**

- `derive` featureによりサブコマンドを宣言的に定義できる
- ヘルプの自動生成・型安全な引数解析が標準で備わっている
- Rustエコシステムでの採用実績が最も多い

## ADR-004: エラーハンドリングとして anyhow を選択

**日付**: 2025年

**ステータス**: 採用

**背景**

rusqlite・clap・stdなど異なるクレートから返る複数のエラー型を統一的に扱う必要があった。

**検討した選択肢**

- 手実装（自前のenum）: 型の変換を全て手書きする必要がある
- `thiserror`: カスタムエラー型の定義に適しているがアプリ層では冗長
- `anyhow`: アプリケーション層向けで全エラー型を統一できる

**決定理由**

- `?` 演算子でどのエラー型も伝播できる
- `.context()` でユーザー向けのエラーメッセージを付加できる
- アプリケーション開発では `anyhow` から始めるのがRustの慣習

**トレードオフ**

- エラーの種別による分岐が必要になった場合は `thiserror` の併用を検討する

## ADR-005: シリアライズとして serde を選択

**日付**: 2025年

**ステータス**: 採用

**背景**

Rustの型とJSON等の外部フォーマット間の変換処理が必要だった。

**決定理由**

- Rustエコシステムの事実上の標準
- `#[derive(Serialize, Deserialize)]` マクロで変換コードが自動生成される
- フェーズ2のAPI実装（JSONレスポンス）でもそのまま使用できる

## ADR-006: モジュール構成の責務分離

**日付**: 2025年

**ステータス**: 採用

**背景**

学習目的のため可読性・責務分離を最優先にしたモジュール設計が必要だった。

**決定内容**

```
src/
  main.rs        # エントリーポイント・CLIの起動のみ
  cli.rs         # clapの定義（引数・サブコマンド）
  db.rs          # SQLite接続・マイグレーション
  model.rs       # ドメイン型の定義（Transaction等）
  repository.rs  # DBへのCRUD操作
  bin/
    api.rs       # APIサーバーのエントリーポイント（フェーズ2）
```

**決定理由**

- 1ファイルに全処理を書くとソースを読んで理解する学習目的に反する
- CLIとAPIでビジネスロジック（repository.rs）を共有できる
- 将来の機能追加時に変更箇所が明確になる

## ADR-007: APIサーバーを別バイナリとして分離

**日付**: 2025年

**ステータス**: 採用（フェーズ2）

**背景**

フェーズ2でHTTP APIを追加するにあたり、CLIとAPIサーバーのバイナリ構成を決定する必要があった。

**検討した選択肢**

- 同一バイナリ（`kakeibo serve` でサーバー起動）: シンプルだがデプロイ対象が大きくなる
- 別バイナリ（`kakeibo-api`）: Cloud Runには最小限のバイナリのみデプロイできる

**決定理由**

- Cloud Run には `kakeibo-api` のみをデプロイすることでデプロイ対象を最小化できる
- CLI固有の処理（clap等）をAPIサーバーに混入させない設計として綺麗
- `repository.rs` / `model.rs` / `db.rs` は共有するためコードの重複は発生しない

## ADR-008: クラウドプラットフォームとして Google Cloud を選択

**日付**: 2025年

**ステータス**: 採用（フェーズ2）

**背景**

スマートフォンからのアクセスを可能にするためAPIサーバーのクラウドデプロイが必要になった。

**検討した選択肢**

- Google Cloud（Cloud Run）: Google認証との親和性が高い
- Azure（Container Apps）: 無料枠が厳しく認証実装コストが高い
- AWS（App Runner）: 認証に Cognito 等の追加実装が必要

**決定理由**

- Android アプリで Google Sign-In を使う方針のため認証がネイティブに統合できる
- Cloud Run の無料枠（月180万リクエスト）が個人利用に十分
- GitHub Actions との連携が公式 Action として整備されている
- リージョンは `asia-northeast1`（東京）を使用

**トレードオフ**

- Google Cloud に対するベンダーロックインが発生する

## ADR-009: 認証方式として Google ID Token を選択

**日付**: 2025年

**ステータス**: 採用（フェーズ2）

**背景**

個人利用のAPIサーバーに対して自分のみがアクセスできる認証が必要だった。

**決定内容**

- Android アプリで Google Sign-In を実施し ID Token を取得
- API リクエストの `Authorization: Bearer <token>` ヘッダーに付与
- サーバー側で Token を検証し `sub` クレームを `user_id` として使用

**決定理由**

- 自前のユーザー管理が不要
- Google Cloud の IAM と統合されており追加のインフラ設定が不要
- `sub` クレームはGoogle アカウントごとに一意であり `user_id` として適切

**トレードオフ**

- Google アカウントへの依存が発生する

## ADR-010: DB を Turso（libsql）に移行

**日付**: 2025年

**ステータス**: 採用（フェーズ2）

**背景**

Cloud Run はコンテナ再起動のたびにファイルが消えるため SQLite のファイルベース運用が不可能になった。

**検討した選択肢**

- Cloud SQL（PostgreSQL）: 無料枠なし・有料
- Supabase（PostgreSQL）: 無料枠あり（500MB）だがスキーマ変更が必要
- Turso（SQLite互換）: 無料枠あり（8GB）・rusqliteとAPI互換性が高い

**決定理由**

- `rusqlite` から `libsql` クレートへの移行で `db.rs` の変更が最小限
- 無料枠（8GB）が個人利用に十分
- SQLite互換のためフェーズ1の資産をそのまま活かせる

## ADR-011: Android UI として Kotlin + Jetpack Compose を選択

**日付**: 2025年

**ステータス**: 採用（フェーズ3）

**背景**

スマートフォンからの入力UIが必要になった。

**検討した選択肢**

- Rust（NDK経由）: 可能だがAndroid UIの構築が極めて困難
- Flutter + Rust（flutter_rust_bridge）: UIはFlutter・ロジックはRustで分担できる
- Kotlin + Jetpack Compose: Androidの標準・UIに集中できる

**決定理由**

- 入力フォームとAPI通信のみの薄いアプリのためビジネスロジックはサーバー側に集約できる
- Kotlin は Android 開発のデファクトスタンダードでドキュメントが充実している
- Jetpack Compose は宣言的UIで学習コストが Flutter に近い

**トレードオフ**

- Rust 以外の言語が追加されるためコンテキストの切り替えが発生する
